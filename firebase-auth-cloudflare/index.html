Firebase Google 登入在 Cloudflare Pages 無反應

部署成功、畫面正常——但點了「用 Google 帳號登入」卻完全沒反應。這篇記錄如何用一句症狀描述讓 AI 快速鎖定根本原因，並順帶評估下一步功能的可行性。

2026-03-17 問題解決型
我用 AI 做了什麼 描述「點 Google 登入沒反應」的症狀，讓 AI 診斷 Firebase 授權域名缺失的根本原因，並取得逐步修復指引
沒有 AI 的話 需要自己翻 Firebase 文件、逐一排查（console 錯誤 → 授權域名 → 防彈窗設定），可能花 30–60 分鐘才能定位問題
最終成果 3 分鐘內取得明確解法：在 Firebase Console 新增 Cloudflare Pages 域名到授權清單即可修復
01

問題描述

⚠ Bug：點 Google 登入按鈕，完全沒反應
!
症狀：旅遊 App 部署到 Cloudflare Pages 後，點擊「用 Google 帳號登入」按鈕，頁面完全沒有任何回應——沒有 popup、沒有跳轉、沒有錯誤訊息。

環境：React + Firebase Authentication（Google 登入），部署在 Cloudflare Pages（例如 travel-app-xxx.pages.dev）。

容易誤判的方向：因為沒有明顯錯誤，容易以為是按鈕的 onClick 沒有綁定，或是瀏覽器 popup 被封鎖——但其實兩者都不是根本原因。

02

根本原因

Firebase Authentication 有一套授權域名白名單機制：只有列在白名單內的域名，才允許發起 OAuth 登入流程。

預設情況下，Firebase 只信任兩個域名：

  • localhost（本機開發用）
  • your-project-id.firebaseapp.com（Firebase Hosting 用）

當你把 App 部署到 Cloudflare Pages（travel-app-xxx.pages.dev），這個域名不在白名單內，Firebase 會靜默拒絕請求——不拋錯、不跳 popup、就是沒反應。

!
這個行為是 Firebase 的安全設計：未授權的域名發起 OAuth 請求時，Firebase 不會給出任何提示（避免洩漏資訊），所以從前端完全看不到錯誤。
03

解決步驟

只需要把 Cloudflare Pages 的域名加入 Firebase 的授權清單，問題即可解決。

  1. 進入 Firebase Console → 選擇你的專案
  2. 左側選單點 Authentication
  3. 上方點 Settings（設定）分頁
  4. 找到 Authorized domains（授權網域）區塊
  5. 點 Add domain
  6. 輸入你的 Cloudflare Pages 網址，例如：travel-app-abc123.pages.dev
  7. 儲存後，回到 Cloudflare Pages 重新整理頁面，再試一次 Google 登入
→
如果你有自訂域名（例如 mytrip.com），除了 pages.dev 那個，也要把自訂域名一起加進去，否則換域名後又會無法登入。
→
驗證方式：修改後回到網頁，點擊 Google 登入，應該會跳出 Google 帳號選擇的 popup。若 popup 被瀏覽器封鎖，通常網址列旁會出現「允許彈出視窗」的提示。
04

延伸討論：自動搜尋並新增景點的可行性

修完登入問題後，順帶評估了一個新功能的可行性：能否讓 App 自動搜尋景點並加入現有行程？結論是完全可行，有三種主要技術路線：

實用首選
Option A

Google Places API

搜尋真實景點資料
  • 資料準確（名稱、地址、評分、照片）
  • 有免費額度，小規模使用成本低
  • 整合方式成熟，文件豐富
  • 超出免費額度後需付費
  • 需申請 API Key 並設定帳單
Option B

Claude / ChatGPT API

AI 智慧推薦景點
  • 自然語言輸入（「加一個拍照景點」）
  • 彈性最大，可理解複雜需求
  • 知識截止日期，無法查詢即時資訊
  • 按 token 計費，成本較難預測
Option C

兩者結合

AI 理解需求 + Places 取真實資料
  • 最完整的使用體驗
  • AI 處理模糊需求，Places 確保資料準確
  • 需串接兩個 API，架構較複雜
  • 兩個 API 都有費用
!
這個功能最終暫時擱置，不在本次開發範圍內。但評估過程確認了技術可行性，未來如有需要可以快速啟動。
05

Takeaway

這個案例展示了什麼

症狀描述型提問的威力：你不需要懂根本原因，只需要把「做了什麼、看到什麼（或沒看到什麼）」描述清楚，AI 就能從現象反推技術原因。「點登入沒反應」這句話足以讓 AI 鎖定 Firebase 授權域名這個非常具體的問題。

同樣的方式也適用於功能可行性評估：用一句話說清楚你想做什麼，AI 可以快速拆解成多個技術路線並比較，讓你在動手前就能看清楚選擇。

可移植性
任何 Firebase + 非 Firebase Hosting 的部署 → 都需要手動新增授權域名（Netlify、Vercel、GitHub Pages 同理）
特殊條件 → 只有使用 Firebase Hosting 部署才會自動完成這個設定
