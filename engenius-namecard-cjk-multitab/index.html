01

需求背景

EnGenius Technologies 需要一套名片自動生成系統，從 Google Sheets 讀取員工資料，批次產生 PDF 名片。系統已有基本骨架（Flask + PyMuPDF），但隨著公司員工分布在多個地區，出現了幾個問題：

  • 前端沒有地區分頁：LA / APAC & India / EU / JP 的員工全部混在一起，難以選取
  • 日文員工名片出現亂碼：長音符號「ー」渲染成「■」
  • 名片背頁是靜態的：各地區的公司名稱和網站應該不同，但系統固定寫死
  • 日本分公司有特殊需求：背頁要附上東京本社地址，但東京本社員工本身不需要
  • JP 地址格式需要斷行：〒xxx-xxxx 格式的日本郵遞區號後面要換行
!
這類「功能擴展」型的任務特別適合用 AI 協作：每個小問題都有明確的技術邊界，AI 可以針對各問題獨立提供解法，人只需要驗證輸出是否符合業務需求。

03

實作步驟

Step 1 — 多地區分頁 UI（Jinja2 namespace 技巧）

問題在於 Jinja2 的 {% for %} 內部無法修改外部變數（作用域限制）。要讓每張卡片在跨越多個 tab 時有正確的全域索引，需要用 namespace 物件：

{% set ns = namespace(offset=0) %}
{% for tab_name, cards in tabs_data.items() %}
  {% for card in cards %}
    <div data-global-index="{{ ns.offset + loop.index0 }}">...</div>
  {% endfor %}
  {% set ns.offset = ns.offset + (cards | length) %}
{% endfor %}
→
{% set ns.offset = ... %} 是 Jinja2 2.10+ 才有的語法。舊版 Jinja2 沒有 namespace，需要在後端預先計算索引後傳給模板。
Step 2 — 日文 CJK 字元渲染修復

Manrope 字體沒有 CJK 字元支援，所有日文字元渲染為 ■。解法是在字元層級偵測 CJK，並切換字體：

def has_cjk(ch: str) -> bool:
    cp = ord(ch)
    return (0x3000 <= cp <= 0x9FFF or
            0xF900 <= cp <= 0xFAFF or
            0xFF00 <= cp <= 0xFFEF)

# 在 TextWriter 逐字渲染時切換字體
for ch, (w, t) in zip(text, widths):
    use_font = (font_cjk if (has_cjk(ch) and font_cjk) else font_latin)
    tw.append(fitz.Point(cursor_x, y), ch,
              font=use_font, fontsize=fontsize)
    cursor_x += w + t

CJK 字元的 tracking（字距）要設為 0，否則 NotoSansJP 會產生奇怪的字距。Latin 字元保持原有 tracking 設定。

Step 3 — PDF 背頁動態更新

背頁的公司名稱與網站要依地區從 Sheet 讀取，並且要先清除原有文字再寫入新內容。清除步驟：

BACK_BG_COLOR = (0.0, 144/255, 212/255)  # EnGenius 藍

def _clear_back_text(page):
    # 取得文字邊界後加上 padding 的矩形
    for block in page.get_text("blocks"):
        rect = fitz.Rect(block[:4]).inflate(2)
        page.add_redact_annot(rect, fill=BACK_BG_COLOR)
    # graphics=0 是關鍵：保留背景圖形（logo），只刪除文字 layer
    page.apply_redactions(graphics=0)
→
graphics=0 參數至關重要：預設值會同時刪除重疊的向量圖形（包括 Logo）。設為 0 只刪除文字 layer，背景圖形完整保留。
Step 4 — 日本分公司背頁東京本社地址

業務規則：日本地區員工若辦公室不在東京本社，背頁要加印東京本社地址（方便聯絡）。判斷方式是檢查員工地址是否含有東京本社郵遞區號：

JP_HQ_POSTAL  = "108-0023"
JP_HQ_ADDRESS = "東京本社 〒108-0023 東京都港区芝浦三丁目6番14号 Expert Tamachi10階"

is_jp_branch = (
    card_data.get("_tab") == "JP"
    and JP_HQ_POSTAL not in card_data.get("address", "")
)

# 依是否顯示東京地址，使用不同的 y 座標
BACK_WEBSITE_Y    = 135.29  # 無東京地址（直接在中間）
BACK_WEBSITE_Y_JP = 104.05  # 有東京地址（網站往上移）
BACK_ADDR_Y       = 135.29  # 東京地址的 y 座標

這些 y 座標是從設計師提供的兩份參考 PDF（田中（福岡）和大西（東京本社））用 PyMuPDF 提取出來的精確數值，不是估算值。

Step 5 — NotoSansJP 字體部署到 Vercel

Vercel Serverless 環境沒有系統字體，需要把 TTF 納入 repo。從 Google Fonts API 下載：

# 先取得 CSS，從中解析出 TTF 的直接下載 URL
curl "https://fonts.googleapis.com/css2?family=Noto+Sans+JP&display=swap" \
  -H "User-Agent: Mozilla/5.0"
# 再用 wget 下載實際 TTF（約 5.1MB）
wget -O NotoSansJP-Regular.ttf "[url-from-css]"
git add NotoSansJP-Regular.ttf
!
5.1MB 的 TTF 對 Vercel 50MB Lambda 限制來說是可接受的（其他資源加起來約 15MB）。如果字體更大，可考慮用 subset 工具裁切只需要的字元範圍。
04

踩坑紀錄

⚠ Bug 1：PDF 背頁舊文字沒有被清除
!
症狀：用 draw_rect 填藍色覆蓋文字，視覺上看起來清除了，但用 page.get_text() 仍能讀到舊文字，PDF Reader 也可能顯示舊內容。

根本原因：draw_rect 只是在文字上方畫一個藍色矩形，文字 layer 還在 PDF 的 content stream 裡。

# ❌ 錯誤做法：只是視覺覆蓋
page.draw_rect(rect, color=BACK_BG_COLOR, fill=BACK_BG_COLOR)

# ✅ 正確做法：真正刪除文字 layer
page.add_redact_annot(rect, fill=BACK_BG_COLOR)
page.apply_redactions(graphics=0)  # graphics=0 保留 logo！
→
預防：在 PDF 編輯中，視覺覆蓋 ≠ 資料刪除。任何要「移除」PDF 內容的操作，都應該用 redaction API 而不是繪圖 API。
⚠ Bug 2：apply_redactions 把 Logo 也刪掉了
!
症狀：背頁 Logo 在執行 redaction 後消失，背頁只剩藍色背景。

根本原因：apply_redactions() 預設參數 graphics=1，會同時刪除與 redact 區域重疊的向量圖形，而 Logo 恰好位於 redact 區域內。

# ✅ 修復：graphics=0 只刪文字，保留圖形
page.apply_redactions(graphics=0)
⚠ Bug 3：東京本社長地址被自動縮小字體
!
症狀：「東京本社 〒108-0023 東京都港区芝浦三丁目6番14号 Expert Tamachi10階」字體從 7pt 被縮成 6.8pt。

根本原因：自動縮放的觸發閾值設為 PAGE_WIDTH * 0.88（約 224pt），長地址文字寬度超過此限制。

⚠ Bug 4：Edit 工具找不到字串
!
症狀：用 Edit 工具修改程式碼時，old_string 找不到匹配，工具回報錯誤。

根本原因：程式碼中有不可見的空白差異（trailing spaces、縮排 tab vs space）。

# 正確流程：先用 Grep 找到行號，再用 Read 確認精確內容
# Step 1
grep -n "target_function" generate_cards.py
# Step 2：Read 確認確切的字串格式
# Step 3：才使用 Edit 工具
→
在複雜的程式檔案中使用 Edit 工具前，一定要先 Read 確認確切內容，不要依賴記憶中的字串。
05

AI vs 人 分工

AI 負責
  • 撰寫 CJK 字元偵測函式（Unicode 範圍查表）
  • 實作字元層級字體切換邏輯
  • 發現 apply_redactions(graphics=0) 的正確用法
  • Jinja2 namespace 跨 loop 計數的實作
  • 從 Google Fonts API 下載 NotoSansJP 的指令
  • ZIP blob 下載的 JS 實作（fetch → blob → URL.createObjectURL）
  • 從參考 PDF 提取精確 y 座標的 Python 腳本
人負責
  • 提供兩份參考 PDF（田中 + 大西）作為背頁佈局依據
  • 決定「東京本社員工不需要額外地址」的業務規則
  • 確認郵遞區號 108-0023 是東京本社的判斷依據
  • 驗證每次修改的輸出 PDF 是否符合預期
  • 決定 NotoSansJP 要納入 repo（而非 runtime 下載）

這個案例的分工模式很典型：業務邏輯判斷由人決定，技術實作細節由 AI 執行。特別是「東京本社地址」的那段，我（人）只說了「非東京分公司要加東京本社地址」，AI 自動把它轉化為「檢查 address 欄位是否含有 108-0023」的具體判斷條件，這種業務語言 → 程式邏輯的轉譯是 AI 協作中最有效的部分。

06

Takeaway

這個案例展示的思路

「症狀描述型」提問在 debug 情境中最有效。每次遇到問題，不是問「我的程式碼哪裡錯了」，而是描述具體症狀：「視覺上看起來清除了但 get_text() 仍能讀到舊內容」。AI 能從症狀直接推斷根本原因（text layer 沒有被實際刪除），提供精確的解法。

參考文件驅動設計：日本背頁佈局問題，AI 初始給的是估算座標，效果不精準。解法是提供真實參考 PDF，讓 AI 寫腳本提取精確座標，再用提取到的數值重建邏輯。「給 AI 原始資料讓它自己測量」比「讓 AI 猜」精準得多。

修 PDF 文字 → 用 add_redact_annot + apply_redactions(graphics=0)，不要用 draw_rect
多語言字體 → 字元層級切換字體，CJK 字元用 NotoSansJP 且 tracking=0
Jinja2 跨 loop → {% set ns = namespace(counter=0) %}，在 loop 內用 ns.counter 修改
Vercel 字體部署 → 5MB 以內的 TTF 直接納入 repo，不依賴系統字體或 CDN
可移植性聲明

可直接複製的情境：任何需要在 Python PDF 操作中清除並重新寫入文字的場景（合約蓋章、名片客製化、報表個人化）；任何需要在單一 PDF 頁面上混合渲染 CJK + Latin 字元的場景。

特殊前提：Jinja2 namespace 技巧需要 Jinja2 2.10+；apply_redactions(graphics=0) 是 PyMuPDF 1.18+ 的參數；NotoSansJP 授權為 SIL Open Font License，可商業使用。
