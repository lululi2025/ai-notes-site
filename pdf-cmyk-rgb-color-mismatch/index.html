01

問題描述

⚠ 症狀：背面文字替換區域與背景色不同
!
環境：Python + PyMuPDF (fitz)，用 add_redact_annot(fill=color) 刪除 PDF 背面舊文字後再寫入新文字。

現象：名片背面（藍底白字）經過文字替換後，redaction 填色區域的藍色跟原始背景的藍色有明顯色差，肉眼可見一個不同顏色的矩形補丁。

填色值：程式設定 BACK_BG_COLOR = (0.0, 144/255, 212/255)，理論上是 #0090d4，但渲染結果是 #1b8fcf。

02

根本原因

追蹤過程：從像素取樣到 content stream 分析

第一步嘗試用 get_pixmap() 取樣整個背面，確認原始背景確實是 RGB(0,144,212)。再嘗試用 get_drawings() 提取背景矩形的填色值，得到 fill=(0.0, 0.5638, 0.8304)，但把這個精確值塞進 redaction fill，渲染結果依然是 #1b8fcf，不是 #0090d4。

問題不在數值精度，而是在色彩空間。於是解析 PDF 的 raw content stream：

# PDF 背面 content stream 的前幾行：
0.95 0.26 0 0 k        ← 設定 CMYK 填色
/Perceptual ri          ← 渲染意圖
/GS0 gs                 ← 圖形狀態
0 0 255.118 153.071 re  ← 畫背景矩形（整頁）
f                       ← 填充

關鍵在第一行的 k 運算子：這是 **CMYK 色彩空間**的設色指令。四個參數分別是 C=0.95, M=0.26, Y=0, K=0。

→
根本原因：模板 PDF 的背景用 CMYK (0.95, 0.26, 0, 0) 繪製，但 PyMuPDF 的 add_redact_annot(fill=...) 和 draw_rect(fill=...) 都使用 RGB 色彩空間。PDF 渲染器在混合 CMYK 背景和 RGB 填色時，會因色彩空間轉換產生肉眼可見的色差。
項目色彩空間值渲染結果
原始背景CMYK0.95 0.26 0 0#0090d4
redact fill（之前）RGB(0, 0.5647, 0.8314)#1b8fcf
draw_rect fillRGB(0, 0.5638, 0.8304)#1b8fcf
CMYK 注入（修復後）CMYK0.95 0.26 0 0#0090d4

不論用哪個 RGB 數值，只要色彩空間不同，渲染結果就會有差異。唯一正確的做法是用跟原始背景相同的 CMYK 色彩空間。

03

解決步驟

Step 1 — 提取模板的 CMYK 色值
# 從 content stream 提取 CMYK 值
# "0.95 0.26 0 0 k" → CMYK(C=0.95, M=0.26, Y=0, K=0)
BACK_BG_CMYK = (0.95, 0.26, 0, 0)
Step 2 — Redact 文字時不填色

把 fill=BACK_BG_COLOR（RGB）改成 fill=False，讓 redaction 只負責刪除文字，不負責填色：

# ❌ 之前：用 RGB 填色，色彩空間不匹配
page.add_redact_annot(rect, fill=BACK_BG_COLOR)

# ✅ 修正：只刪除文字，不填色
page.add_redact_annot(rect, fill=False)
page.apply_redactions(graphics=0)  # graphics=0 保留 Logo
Step 3 — 注入 CMYK 指令補回背景

用 PyMuPDF 的低階 API fitz.TOOLS._insert_contents() 直接在 page 的 content stream 裡插入 CMYK 繪圖指令：

c, m, y_c, k = BACK_BG_CMYK  # (0.95, 0.26, 0, 0)

# 組合 CMYK 矩形指令
cmyk_parts = [f"q {c} {m} {y_c} {k} k"]  # q = 保存狀態, k = 設 CMYK 填色
for r in redact_rects:
    # PyMuPDF 座標 (y↓) → PDF 原生座標 (y↑)
    pdf_y = page_h - r.y1
    cmyk_parts.append(
        f"{r.x0:.2f} {pdf_y:.2f} {r.width:.2f} {r.height:.2f} re"
    )
cmyk_parts.append("f Q")  # f = 填充, Q = 恢復狀態

# 注入到 content stream（1 = append 到尾端）
fitz.TOOLS._insert_contents(
    page, " ".join(cmyk_parts).encode(), 1
)
→
座標轉換注意：PyMuPDF 的座標系 y 軸往下，PDF 原生座標 y 軸往上。插入 raw content stream 時要做 pdf_y = page_height - pymupdf_y1 的轉換。
04

驗證方式

生成名片後用 get_pixmap(dpi=300) 對背面密集取樣，比對每個點位是否為 RGB(0,144,212)：

pix = back_page.get_pixmap(dpi=300)
target = (0, 144, 212)

for label, fx, fy in sample_points:
    x, y = int(w * fx), int(h * fy)
    r, g, b = pix.pixel(x, y)[:3]
    ok = 'OK' if (r, g, b) == target else 'DIFF'
    print(f'{label} → RGB({r},{g},{b}) {ok}')

05

預防與通則

→
通則：修改 PDF 時，先確認原始色彩空間。設計工具（Illustrator、InDesign）輸出的印刷品 PDF 幾乎一定是 CMYK。用 PyMuPDF 的 RGB API（draw_rect、add_redact_annot）操作這類 PDF，一定會遇到色差。
情境原始色彩空間建議做法
網頁生成的 PDF通常 RGBadd_redact_annot(fill=...) 可直接用
設計稿 / 印刷品 PDF通常 CMYK解析 content stream 取得 CMYK 值，用 _insert_contents 注入
混合色彩空間 PDF可能同時有逐頁檢查，分別處理
如何快速判斷 PDF 色彩空間
# 讀取 content stream，搜尋色彩運算子
content_bytes = doc.xref_stream(page.get_contents()[0])
text = content_bytes.decode('latin-1')

if ' k ' in text or text.strip().endswith(' k'):
    print("CMYK（k = 非描邊 CMYK 設色）")
if ' rg ' in text or ' RG ' in text:
    print("RGB（rg/RG = RGB 設色）")
if ' g ' in text or ' G ' in text:
    print("Grayscale（g/G = 灰階設色）")
06

Takeaway

這個案例展示的思路

「截圖驅動 debug」：使用者只提供了一張「顏色不一樣」的截圖，沒有錯誤訊息，沒有 stack trace。AI 從截圖觀察出發，撰寫像素取樣腳本、解析 PDF content stream、測試不同色彩空間的渲染結果，系統性地追蹤到根本原因。

這種「視覺 bug」的 debug 流程：觀察 → 量化（像素取樣）→ 溯源（content stream）→ 假設（色彩空間）→ 驗證（對照測試）→ 修復。每一步都可以要 AI 寫一次性的驗證腳本。

核心知識 → PDF 中 k = CMYK 設色，rg = RGB 設色，混用會色差
關鍵 API → fitz.TOOLS._insert_contents(page, bytes, 1) 注入原生 PDF 指令
座標轉換 → PyMuPDF y↓ → PDF y↑：pdf_y = page_h - pymupdf_y1
可移植性聲明

可直接複製的情境：任何用 PyMuPDF 對「設計稿 PDF」做文字替換、浮水印、蓋章時遇到的色差問題。只要原始 PDF 是 CMYK 色彩空間（設計師用 Illustrator/InDesign 輸出的幾乎都是），PyMuPDF 的 RGB 填色就會色差。

特殊前提：fitz.TOOLS._insert_contents() 是 PyMuPDF 的內部 API（前綴底線），未來版本可能變更。目前 PyMuPDF 1.18~1.24 可用。
