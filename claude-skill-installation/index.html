01

什麼是 Claude Skill

Claude Skill 是一種模組化的擴充能力，透過一組結構化的指令、腳本和資源檔，教會 Claude 處理特定任務。可以把它想成「外掛」或「食譜」— 它不是給 Claude 新工具（那是 MCP 的工作），而是教 Claude 怎麼用已有的工具去完成特定工作流程。

舉例來說，一個 sync-context skill 會教 Claude：當使用者說「同步進度」時，去掃描 git log、檢查目錄結構、讀取現有的 CLAUDE.md，然後生成或更新一份專案交接文件。

→
一個好的類比：MCP 是廚房裡的鍋碗瓢盆（工具），Skill 是食譜（怎麼用工具做出一道菜）。兩者可以搭配使用，但很多時候只有 Skill 就夠了。
02

Skill 的核心結構

不管在哪個平台，Skill 的核心結構都是一樣的。最小可用的 skill 就是一個資料夾裡放一個 SKILL.md 檔案：

skill-name/
├── SKILL.md ← 必要，主指令檔
└── 附加資源（選用）
    ├── scripts/   ← 可執行腳本
    ├── references/ ← 參考文件
    └── assets/    ← 模板、圖片等
SKILL.md 的組成

SKILL.md 包含兩個部分：YAML frontmatter（名稱 + 觸發描述）和 Markdown 本體（具體指令）。frontmatter 的 description 欄位至關重要 — 它決定了 Claude 什麼時候會觸發這個 skill。

三層漸進式載入
層級內容何時載入大小建議
Metadataname + description永遠在 context 中~100 words
SKILL.md 本體完整指令觸發時才載入<500 行
附加資源scripts, references, assets需要時才讀取無限制
!
這意味著你可以裝很多 skill 而不會拖慢 Claude — 每個 skill 平時只佔約 100 tokens 的 metadata。只有真正用到的 skill 才會完整載入。
03

四個平台安裝比較

這是最容易搞混的部分。Claude 有四個不同的產品，都支援 Skill，但安裝方式不同：

平台安裝格式安裝位置特色
Claude.ai（網頁版）.skill 檔（ZIP）Settings → Customize → Skills透過 UI 上傳，跨 session 保留
Claude Code（CLI）資料夾 + SKILL.md.claude/skills/ 目錄可 git 追蹤，團隊共享
Claude Desktop（桌面 App）.skill 檔（ZIP）App Settings 內上傳同 Claude.ai 格式，可搭配 MCP
Claude API（程式化）ZIP / 字串 / skill_idAPI 請求的 container 參數最彈性，支援多種格式
Claude Code 的路徑層級
路徑作用範圍適用情境
~/.claude/skills/skill-name/個人全域，所有專案可用通用工具型 skill
./project/.claude/skills/skill-name/僅限該專案專案特定的 conventions、workflows
→
最容易搞混的點：Claude.ai 和 Claude Desktop 都用 .skill（ZIP）格式，Claude Code 用資料夾。如果有人說「Claude 的 skill 要打包成 ZIP」，他們說的是 Claude.ai / Desktop，不是 Claude Code。
04

實作：安裝 sync-context skill

這次對話中，我們實際操作了一個 sync-context skill 的安裝。這個 skill 的功能是在多次 AI session 之間保留專案進度，透過維護一份 CLAUDE.md 檔案來當作交接文件。

在 Claude.ai 網頁環境中的嘗試
# 在 Claude.ai 對話環境中執行
mkdir -p /mnt/skills/user/sync-context
cp /mnt/user-data/uploads/SKILL.md /mnt/skills/user/sync-context/SKILL.md

指令成功了，檔案也確實出現了。但關鍵問題是：/mnt/skills/user/ 是 Claude.ai 雲端環境中的暫存檔案系統，session 結束就會重置。所以這個「安裝」是暫時的。

打包成 .skill 檔

為了永久安裝到 Claude.ai，需要打包成 .skill 檔（ZIP 格式），然後透過 Settings UI 上傳。

這個 skill 該裝在哪？

因為 sync-context 的核心功能是讀寫本機檔案和執行 git 指令，它最適合放在 Claude Code 的 .claude/skills/ 目錄。在 Claude.ai 網頁版雖然能觸發，但因為沒有持久的本機檔案系統，寫出來的 CLAUDE.md 下次 session 就消失了。

選哪個平台？
├── 需要讀寫本機檔案？
│   ├── 是 → Claude Code（最佳）或 Claude Desktop + MCP
│   └── 否 → Claude.ai 就夠了
└── 需要團隊共享？
     ├── 是 → Claude Code（放專案 repo 裡）
     └── 否 → 任一平台皆可
05

常見踩坑點

⚠以為在 Claude.ai 對話中安裝就是永久安裝
!
症狀：用 cp 把 SKILL.md 放到 /mnt/skills/user/，看起來成功了，但下次開新對話 skill 就不見了。

根本原因：Claude.ai 的對話環境是暫存的，每次 session 都會重置。要永久保留，必須透過 Settings UI 上傳 .skill 檔。

→
預防：記住「在對話中操作檔案 ≠ 永久安裝」。永久安裝一律透過 UI 或本機檔案系統。
⚠ZIP 打包格式錯誤
!
症狀：上傳 .skill 檔後，Claude 無法讀取或觸發。

根本原因：ZIP 根目錄必須是 skill 的資料夾名稱，不能直接把 SKILL.md 放在 ZIP 根目錄。

→
預防：壓縮時選「壓縮資料夾」而非「壓縮資料夾裡的檔案」。或用打包腳本自動處理。
⚠Skill 裝了但不觸發
!
症狀：安裝了 skill 但怎麼問都不觸發。

根本原因：YAML frontmatter 的 description 寫得太模糊，或問題太簡單，Claude 認為不需要 skill。

→
預防：description 要寫得具體且「推進」— 明確列出觸發短語，例如「Use this skill whenever the user says '同步進度', '更新 CLAUDE.md'...」
06

AI 協作回顧

這次對話展示了一個典型的「從實作中學概念」的 AI 使用模式。不是先讀完所有文件再動手，而是直接嘗試安裝一個 skill，在碰壁的過程中自然地學到了完整知識。

AI 負責
  • 解讀 SKILL.md 的用途和結構
  • 嘗試在對話環境中安裝 skill
  • 搜尋官方文件整理平台差異
  • 打包 .skill 檔案
  • 產出視覺化比較圖
人負責
  • 上傳 SKILL.md 檔案
  • 提出關鍵問題推動探索
  • 糾正 AI 的錯誤認知
  • 判斷 skill 的適用場景
關鍵轉折點

最重要的推進來自使用者的一個糾正：當 AI 說「我沒辦法幫你安裝」時，使用者反問「你不是可以用 copy to skill 嗎？」— 這促使 AI 實際嘗試了指令，結果成功了。這引出了更深的問題：安裝成功 ≠ 永久生效，進而觸發了對暫存 vs 永久檔案系統的完整討論。

提問模式：層層拆解型

這次對話的提問節奏：先從「這是什麼」開始 → 「怎麼安裝」→ 碰到問題後追問「為什麼不行」→ 最後擴展到「不同地方有什麼差異」。每一步都建立在前一步的理解之上。

07

Takeaway

核心收穫

這個案例展示了什麼：透過「邊做邊問」的方式用 AI 學習新工具，比先讀完文件再動手更有效率。AI 在嘗試過程中犯的錯（以為暫存安裝是永久的）反而成了最好的教學素材。

可移植性：這個「從實作出發」的學習模式適用於任何新工具或新框架。上傳你手邊有的檔案，讓 AI 解釋它、嘗試用它、在過程中自然學到周邊知識。特殊條件是需要 Claude.ai 的 Code Execution 功能開啟。

如果你也要學一個新的 Claude Skill 概念，先問 AI：

起手式 1 → 「我有一個 SKILL.md 檔案，幫我解釋它在做什麼，然後告訴我該怎麼安裝到我的環境」
起手式 2 → 「我用的是 [Claude Code / Claude.ai / Desktop]，幫我把這個 skill 打包成正確的格式」
