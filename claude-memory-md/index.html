01

任務背景

Claude Code 有一個叫做 auto memory 的機制：它會在專案資料夾對應的位置維護一個 MEMORY.md，裡面記錄的東西會在每次對話開始時被自動讀入。這相當於給 AI 一份「這個專案我需要知道的事」備忘錄。

這個 ISC West 2026 的展覽網站專案，原本的 MEMORY.md 只有兩段：一段是 ISC West 的基本路徑和 GitHub 連結，另一段是另一個完全不同的專案（ai-notes-site）的資訊被混在一起。

問題在於：這份記憶太薄。AI 每次重新開啟對話時，不知道這個專案有哪些坑，也不知道展示內容是什麼。當出現模糊指令（例如「整理 claude.md」）時，AI 也沒有足夠的背景知識主動判斷意圖。

→
核心概念：MEMORY.md 不只是給人看的備忘錄，而是給 AI 自己讀的「工作前準備」。寫得越結構化、越包含技術細節，AI 下次工作的品質就越高。
02

AI 使用策略

提問模式：意圖委派型

這次的提問非常短：「整理 claude.md」。這是一種意圖委派型提問——只說目標，不說方法，讓 AI 自己判斷怎麼做。這種方式適合 AI 對專案已有基本認識的場景；如果 AI 不確定，它會主動釐清。

關鍵策略：主動讀取相關檔案再下手

在修改 MEMORY.md 之前，AI 先自行讀取了 IDE 中已開啟的 ISC West Demo Script.md。這讓 AI 在整理記憶時，能加入真實的 Demo 內容概覽，而不只是複製貼上原本的資訊。

這是一個值得複製的模式：整理「記憶文件」之前，先讀一份有代表性的內容文件，這樣 AI 能從中萃取有用資訊填入記憶。

釐清歧義，不盲目執行

原始指令「整理 claude.md」有歧義（找不到 CLAUDE.md 檔案，但有 MEMORY.md）。AI 搜尋後發現沒有目標檔案，主動提出三個選項讓使用者確認，而不是猜測後直接動手。這避免了做錯方向的風險。

03

關鍵對話轉折

步驟 發生什麼事 為什麼重要
使用者輸入「整理 claude.md」 指令模糊，目標檔案不存在 測試 AI 能不能不猜測、主動釐清
AI 搜尋後發現沒有 CLAUDE.md AI 提出三個可能的目標讓使用者選擇 避免盲目執行錯誤方向
使用者選「整理全域 MEMORY.md」 方向明確，AI 開始讀取現有 MEMORY.md 這個選擇決定了整個工作方向
AI 主動讀取 Demo Script.md IDE 中已開啟此檔案，AI 判斷這是相關內容 讓整理後的 MEMORY.md 能包含 Demo 概覽
AI 重寫 MEMORY.md 並存檔 分四個子區塊，加入技術注意事項 AI 自己改善了自己未來的工作環境
!
這整個過程只花了 1–2 分鐘。短指令 → 釐清 → 執行，這是使用 Claude Code 的高效模式：不需要給出詳細規格，讓 AI 在必要時主動問，然後快速完成。
04

整理前後對比

以下是 MEMORY.md 的核心差異。重點不在篇幅增加，而在資訊密度與結構：

Before — 整理前（15 行）
## ISC West 2026 專案
- 本機路徑：`d:\00. Marketing\...`
- GitHub：https://github.com/...(branch: main)
- Cloudflare：Direct Upload 模式，需手動上傳
- 主要檔案：index.html（~2MB）、booth.html（~2.9MB），
  含 base64 圖片，Edit 工具無法直接改，需用 Python 腳本
After — 整理後（37 行）
## ISC West 2026 專案

### 基本資訊
- 本機路徑、GitHub、Cloudflare（同前）

### 主要檔案
- index.html（~2MB）、booth.html（~2.9MB）
- **兩個檔案含大量 base64 圖片，Edit 工具無法直接修改，需用 Python 腳本**
- ISC West Demo Script.md：現場 Demo 腳本（6 個 Demo + Q&A）
- ISC West Bullet Points.docx：產品賣點整理

### 技術重點
- i18n 字串不能含單引號（曾導致 JS 語法錯誤，用 &#39; 或雙引號包覆）
- iframe 語言同步：需在 iframe load 後再發送語言訊息
- 密碼保護：Vercel Edge Middleware

### Demo 內容概覽（6 個 Demo）
1. AI 情境分析 → 2. 姿勢偵測 + Webhook → 3. GenFace
4. AI 自然語言搜尋 → 5. Share QR Code → 6. AI-NVS

最關鍵的新增是「技術重點」子區塊：這兩條紀錄（i18n 單引號問題、iframe 語言同步）都是這個專案過去踩過的坑，被寫進記憶後，AI 日後工作就不會再犯同樣的錯。

05

AI vs 人 分工回顧

AI 負責
  • 搜尋專案目錄，確認目標檔案是否存在
  • 提出三個候選方向，供使用者選擇
  • 讀取現有 MEMORY.md，分析資訊缺口
  • 主動讀取 Demo Script.md，萃取有用資訊
  • 決定新的文件結構（四個子區塊）
  • 從 git commit history 提取已知技術問題
  • 撰寫並存檔新版 MEMORY.md
人負責
  • 發出一句模糊指令「整理 claude.md」
  • 從三個選項中確認目標（MEMORY.md）
  • （無需其他操作，直接等 AI 完成）
→
這是「最小授權」模式：人只做決策，AI 做所有研究與執行。關鍵是 AI 在不確定時主動問，而不是猜測——這讓模糊指令也能安全地運作。
06

Takeaway

這個案例展示了什麼

AI 可以主動維護自己的工作環境。MEMORY.md 是 AI 的「工作前準備清單」，定期讓 AI 更新它，相當於每次工作前先做交接會議——AI 會記得上次的坑，也知道這個專案的獨特限制。

更重要的是：這件事本身只需要一句話。你不需要告訴 AI 要加什麼、怎麼組織，AI 會讀原始內容、讀相關文件，然後自己判斷怎麼補充。

可移植性
適合直接複製的情境 → 任何有使用 Claude Code 的專案，只要有 MEMORY.md 就能套用
特殊前提條件 → 需要有「過去踩過坑的歷史」可以讓 AI 萃取（git commits、已開啟的文件）
不適用的場景 → 全新專案，沒有任何歷史脈絡可讀，此時只能手動提供初始記憶
如果你也要做，先問 AI 什麼

起手式 1（定期維護）：

幫我整理這個專案的 MEMORY.md，
讀一下最近的 git commits 和現在開著的檔案，
看看有什麼技術細節或已知問題值得記錄進去。

起手式 2（剛解完一個 bug）：

剛才這個問題解決了，幫我把根本原因和解法
寫進 MEMORY.md 的「技術重點」區塊，
避免下次再踩同一個坑。
