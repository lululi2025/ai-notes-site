01

需求背景

ISC West 是美國最大的安防展覽，每年在拉斯維加斯舉辦。2026 年的攤位需要一個專屬展示網站，讓現場工作人員能在平板或電腦上導覽產品介紹，同時支援中英文雙語切換（服務不同語言的訪客）。

網站不需要後端，也不需要 CMS，純靜態即可。圖片素材量很大，採用 base64 內嵌方式處理，讓頁面完全自包含，不依賴外部圖片伺服器。

→
展覽網站的特殊需求：現場 Wi-Fi 不穩定，因此網站必須能離線運作，所有資源（圖片、字型）都要內嵌或預載。base64 圖片雖然讓檔案很大，但確保了完全離線可用。
02

靜態 HTML（無框架） 展覽環境不需要動態功能，純靜態最穩定、部署最簡單

03

實作步驟

Step 1｜建立兩個主要頁面

網站包含兩個主要頁面：index.html（產品主展示頁）和 booth.html（攤位介紹頁）。所有圖片素材以 base64 格式內嵌，讓頁面完全獨立。這讓每個 HTML 檔案都非常大（~2–3 MB），需要注意：這個大小讓 Claude Code 的 Edit 工具無法直接修改，所有批次編輯需要用 Python 腳本處理。

Step 2｜加入雙語切換系統

在 HTML 元素上加入 data-i18n 屬性，值對應一個語言 key。JavaScript 讀取這些屬性，根據目前語言切換對應的文字內容。

<!-- HTML 標記語言 key -->
<h1 data-i18n="hero.title"></h1>

<!-- JS 切換函式 -->
function applyLanguage(lang) {
  document.querySelectorAll('[data-i18n]').forEach(el => {
    const key = el.dataset.i18n;
    el.textContent = i18n[lang][key] || key;
  });
}

頁面內有 iframe 嵌入其他子頁，語言切換時需同步通知 iframe，使用 postMessage 跨頁通訊。

Step 3｜加入密碼保護

在部署前加入 Vercel Edge Middleware，要求輸入密碼才能瀏覽網站。這讓展覽前的準備階段不會對外公開，也能控制誰可以存取 Demo 頁面。

Step 4｜部署到 Cloudflare Pages

使用 Cloudflare Pages 的 Direct Upload，直接上傳 HTML 檔案，幾分鐘內就能取得公開網址。

04

踩坑紀錄

⚠ Bug 1：i18n 字串含單引號導致 JS 語法錯誤
!
症狀：雙語切換功能完全失效，Console 顯示 SyntaxError: Unexpected token，但找不到明顯問題。

根本原因：i18n 語言物件直接寫在 .js 或 inline <script> 中，其中某條字串含有英文單引號（例如 don't），而外層用的是單引號包覆 → 提前截斷字串，造成語法錯誤。

// ❌ 錯誤：外層單引號 + 字串含單引號
const i18n = {
  en: { desc: 'It don't require setup' }
}

// ✅ 修正方式一：改用 HTML entity
const i18n = {
  en: { desc: 'It don't require setup' }
}

// ✅ 修正方式二：外層改用雙引號
const i18n = {
  en: { desc: "It don't require setup" }
}
→
預防：i18n 字串統一用雙引號包覆，或在建立語言物件時直接用 Template Literal（`...`），避免引號衝突。
⚠ Bug 2：iframe 語言切換不同步
!
症狀：切換語言後，主頁面文字正確切換，但 iframe 內的子頁文字仍停留在原語言。

根本原因：語言切換事件在 iframe 尚未載入完成時就發送了 postMessage，訊息被丟棄。iframe 的 load 事件尚未觸發，還沒開始監聽訊息。

// ❌ 錯誤：直接發送，不管 iframe 是否已載入
function switchLang(lang) {
  applyLanguage(lang);
  iframe.contentWindow.postMessage({ lang }, '*');
}

// ✅ 修正：等 iframe load 完成再發送
iframe.addEventListener('load', () => {
  iframe.contentWindow.postMessage({ lang: currentLang }, '*');
});

// 切換時也要補發
function switchLang(lang) {
  currentLang = lang;
  applyLanguage(lang);
  // iframe 已載入才發，否則 load 事件會補發
  if (iframeLoaded) {
    iframe.contentWindow.postMessage({ lang }, '*');
  }
}
→
規則：任何跨 iframe 的 postMessage 都要確認 iframe 已觸發 load 事件後才發送。可以用一個 flag 記錄 iframe 是否已就緒。
05

部署方式

這個專案使用 Cloudflare Pages Direct Upload，而非 Git 整合部署。原因是展覽素材（base64 圖片）讓 HTML 檔案非常大，加上修改頻率不高，手動上傳反而更直接。

項目細節
平台 Cloudflare Pages
部署方式 Direct Upload（手動上傳 HTML 檔案）
GitHub 整合 無（Cloudflare 和 GitHub 各自獨立）
更新流程 修改本機 HTML → Cloudflare 後台手動上傳
密碼保護 Vercel Edge Middleware（展覽前限制存取）
!
由於 HTML 檔案含 base64 圖片（單檔 2–3 MB），Claude Code 的 Edit 工具會超時，無法直接修改這兩個檔案。需要用 Python 腳本做批次字串替換，或在 Claude Code 外部用文字編輯器修改。
06

Takeaway

這個案例展示了什麼

展覽網站有獨特的限制：要能離線運作、要輕量部署、要快速迭代。靜態 HTML + base64 圖片 + Cloudflare 是一個可行的組合，但要做好心理準備——大型靜態 HTML 的維護成本比看起來高，特別是使用 AI 工具輔助修改時。

兩個 bug 都是時序問題：一個是 JS 字串解析的時序，一個是 iframe 載入的時序。這類 bug 的共同特徵是「功能本身沒問題，只是在錯誤的時間點執行」。

可移植性
適合直接套用 → 任何需要離線運作的靜態展示網站：展覽、簡報、產品 Demo
base64 圖片策略 → 圖片少（<10 張）才適合，多的話應改用外部 CDN + Service Worker 離線快取
雙語切換踩坑 → i18n 字串用雙引號或 Template Literal；iframe 同步要等 load 事件
